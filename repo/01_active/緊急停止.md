# 緊急停止

最終更新：2026-05-07
目的：本番アカウント (@${THREADS_USERNAME}) への意図しない Threads 投稿を、Discord から 1 コマンドで止める／復旧するための設計と運用手順をまとめる。

> 構成全体像は `07_構成図.md` 参照。本ファイルは緊急停止機能だけにフォーカス。

---

## 1. 設計

### 1.1 何を止めるか（スコープ）

| 対象 | 緊急停止中の挙動 |
|---|---|
| `publish.py --profile production`（朝/昼/夜の本番投稿） | **冒頭ガードで即 exit**（Threads API を呼ばない） |
| `publish.py --profile test` | 影響なし（テストアカウント `@${THREADS_TEST_USERNAME}` には流れる） |
| `generate.sh`（5:00 の 5 案生成） | 影響なし |
| `fetch_metrics.py`（翌 7:00 の反応値取得） | 影響なし（`status==published` 限定なので、停止中に skip された slot は集計対象外） |
| Discord Bot（承認・修正・照会・ドラフト・バックログ） | 影響なし |
| `state.json` の更新 | 影響なし（`state_update.py` / `revise.py` / `add_seed.py` は通常動作） |

**設計意図**：「投稿だけ止めて、生成・承認フローは通常通り回す」。停止中も翌日分の 5 案生成・承認は進められるので、復旧後すぐ通常運用に戻せる。

### 1.2 仕組み（センチネルファイル方式）

```mermaid
flowchart LR
    M[OWNER] -->|/緊急停止 [理由]| BOT[Bot 常駐]
    BOT -->|--on| ES[emergency_stop.py]
    ES -->|touch| KS[(state/EMERGENCY_STOP_PRODUCTION)]

    LD[launchd 7/12/20時] --> PB[publish.py<br/>--profile production]
    PB -->|起動冒頭で<br/>存在チェック| KS
    KS -->|存在| EX[sys.exit 0<br/>+ Discord 通知]
    KS -.->|不在| TG[Threads Graph API<br/>通常投稿]

    BOT2[OWNER] -->|/緊急停止解除| BOT
    BOT -->|--off| ES
    ES -->|rm| KS

    style KS fill:#ffe7a3
    style EX fill:#ffd0d0
    style TG fill:#c8f0c8
    style ES fill:#ffd5c2
```

- センチネルファイル `state/EMERGENCY_STOP_PRODUCTION` の **存在 / 不在** だけで状態管理（プロセス間通信なし、再起動を跨いでも保持）
- `publish.py` 冒頭にガード 5〜7 行を追加。`--profile production` のときだけチェック
- ガード発動時は **`sys.exit(0)`**（launchd 上は成功扱い、毎回 fail 通知が来ない）+ Discord に「⛔ 緊急停止中のため SLOT 投稿をスキップしました」を 1 通

### 1.3 センチネルファイルの中身

```text
created_at: 2026-05-07T13:42:00+09:00
actor: discord
message_id: 1234567890
reason: 5 案で誤った URL が混入していた、念のため停止
```

- フォーマットは plain text（key: value 行）。Python 側は存在チェックさえできれば中身は使わない
- `--reason` を渡さなければ `reason: (未指定)`
- Discord 経由なら `actor: discord` + `message_id`、CLI 直叩きなら `actor: cli`

### 1.4 採用しなかった案

| 案 | 不採用理由 |
|---|---|
| `switch_profile.sh production → test` 自動切替 | 本番アカウントは保護されるが、test 側に意図せず投稿が積まれる。後で削除作業が必要になる |
| `launchctl bootout` で publish 系 plist 停止 | 効果は最強だが、Discord から `bootstrap` で戻す経路を作るのが複雑（permission 問題も） |
| jsonl で skip ログ記録（案 Y） | 緊急停止の発生頻度が低いため、最初は無し。必要になれば後付け |

## 2. 構成要素

### 2.1 `scripts/emergency_stop.py`（新規・薄い CLI）

| サブコマンド | 動作 |
|---|---|
| `--on [--reason "..."] [--message-id ID] [--actor {discord,cli}]` | センチネル touch（既に存在すれば警告のみ・上書きしない＝最初の停止理由を保持） |
| `--off` | センチネル rm（存在しなければ警告のみ） |
| `--status` | 存在チェック + 中身表示。exit code は存在=0 / 不在=1 |

- `--actor` のデフォルトは `cli`（Bot 経由は `--actor discord` を明示）
- 出力は **Bot がそのまま Discord に貼れる短文**（`✅ 緊急停止を ON にしました` 等）
- 冪等：`--on` を 2 回叩いても 2 重作成しない（最初の理由保持）／`--off` を 2 回叩いてもエラーにしない

### 2.2 `publish.py` のガード追加（5〜7 行）

```python
# 冒頭、args.profile / args.slot を確定した直後に追加
if args.profile == "production":
    killswitch = REPO_ROOT / "state" / "EMERGENCY_STOP_PRODUCTION"
    if killswitch.exists():
        notify_discord(
            f"⛔ 緊急停止中のため {args.slot} 投稿をスキップしました\n"
            f"```\n{killswitch.read_text()}\n```"
        )
        sys.exit(0)
```

- `--dry-run` でも guard は効く（dry-run でもチェックを通すことで動作確認に使える）
- Discord 通知失敗時も `sys.exit(0)` は実行する（投稿しないことが最優先）

### 2.3 `CLAUDE.md` のコマンド表に 3 行追加

| 入力 | 処理 |
|---|---|
| `/緊急停止 [理由]` | `emergency_stop.py --on --reason "..." --actor discord --message-id <id>` |
| `/緊急停止解除` | `emergency_stop.py --off --actor discord --message-id <id>` |
| `/緊急停止状況` | `emergency_stop.py --status` |

- 受付チャンネル：`/予約確認` と同じ 3 つ（`#threads_auto` / 照会専用 / DM）
- 「未対応のコマンド」「自然言語パースに fallback しない」の挙動も `/予約確認` と同じ
- 受付チャンネル外（`#threads_auto_drafts` / `#threads_auto_backlog`）では無視

## 3. 利用方法

### 3.1 通常時：Discord から停止する

```
/緊急停止 5 案で URL ミスを発見
```

Bot 返信例：

```
🛑 緊急停止 ON にしました（理由：5 案で URL ミスを発見）
本番投稿（--profile production）は次回発火からスキップされます。
解除：/緊急停止解除
状況確認：/緊急停止状況
```

理由は省略可（`/緊急停止` だけでも OK）。
受付チャンネルは `#threads_auto` / 照会専用 / DM の 3 つ。

### 3.2 状況確認

```
/緊急停止状況
```

Bot 返信例（停止中）：

```
🛑 緊急停止：ON
created_at: 2026-05-07T13:42:00+09:00
actor: discord
message_id: 1234567890
reason: 5 案で URL ミスを発見
```

Bot 返信例（通常時）：

```
✅ 緊急停止：OFF（本番投稿は通常通り動作中）
```

### 3.3 ターミナルから直接（Discord 経由できない時のバックアップ）

```bash
# 停止
python3 __REPO_ROOT__/repo/01_active/scripts/emergency_stop.py \
  --on --reason "緊急対応"

# 状況確認
python3 .../scripts/emergency_stop.py --status

# 解除
python3 .../scripts/emergency_stop.py --off
```

Mac へ ssh / 物理アクセスできる前提。Bot や Discord が落ちている時の最後の手段。

## 4. 復旧手順

### 4.1 解除コマンド

```
/緊急停止解除
```

Bot 返信例：

```
✅ 緊急停止を解除しました（次回 launchd 発火から本番投稿が復活します）
停止中にスキップされた slot がある場合は手動再送：
python3 scripts/publish.py --slot SLOT --date YYYY-MM-DD --profile production
```

これで `state/EMERGENCY_STOP_PRODUCTION` が消え、次の 7:00 / 12:00 / 20:00 の launchd 発火から `publish.py` のガードを通り抜けて通常投稿される。Mac 再起動や `launchctl` 操作は不要。

### 4.2 自動復旧の挙動

| シナリオ | 挙動 |
|---|---|
| 7:00 投稿前に `/緊急停止` → 7:00 発火が skip → 8:00 に `/緊急停止解除` | 12:00 / 20:00 は通常通り送られる。**7:00 分は送られたまま消える** |
| 7:00 投稿前に `/緊急停止` → 翌日 5:00 まで何もしない | 7:00 / 12:00 / 20:00 全部 skip される。翌日 5:00 の generate.sh は通常通り走り、新しい 5 案が出る |
| 解除し忘れて 24h 経過 → fetch_metrics は対象 slot をどう扱うか | `metrics_24h` は `status==published` の slot しか対象にしないので、skipped slot は無視される（影響なし） |

### 4.3 過去スロットの手動再送

「停止中にスキップされた slot を、解除後に遡って送りたい」場合：

```bash
# 例：2026-05-08 の昼を後追いで送る
python3 __REPO_ROOT__/repo/01_active/scripts/publish.py \
  --slot noon \
  --date 2026-05-08 \
  --profile production
```

- state は `approved` のまま残っているので、これだけで通常通り送れる
- `--date` は **publish される日**（state ファイル名）。前日朝に 5:00 generate された分なら `--date <昨日>`
- 複数 slot を送りたければ `--slot morning` `--slot evening` を別コマンドで叩く（並列実行は禁止：state.json の排他更新は 1 プロセスのみ）

### 4.4 送らない選択

スキップされた slot を **送らないまま放置する** のも正常な選択。

- state.json では `slot.status` が `approved` のまま残るが、`metrics_24h` 集計対象外なので副作用なし
- 翌日の `generate.sh` は **新しい日付の state** を作るので、旧日付の `approved` 残骸が次の生成に干渉することはない
- `learning/edits/*.md` も発生しない（publish 成功時にだけ書かれる）

## 5. やってはいけないこと

- **手書きで `state/EMERGENCY_STOP_PRODUCTION` を作る・消す**（必ず CLI 経由。手動操作だと `actor` / `message_id` が記録されず、後から「誰が止めた？」が追えない）
- **緊急停止中に `publish.py --profile production` を手動実行**（guard で死ぬので無害だが、混乱の元）
- **解除直後に `launchctl kickstart` で即時発火**（次の定刻発火を待つ。手動再送が必要なら 4.3 の `publish.py` 直叩きを使う）
- **緊急停止を「投稿延期」目的で使う**（投稿延期は `--skip-slot` の責務。緊急停止はあくまで runaway 防止）
- **複数日に渡って解除し忘れる**（毎日の publish 通知が来なくなるので異常検知できる、はずだが念のため `/緊急停止状況` を定期確認）

## 6. 動作確認

実装後に最低限通すチェック：

```bash
# 1. 停止 ON
python3 scripts/emergency_stop.py --on --reason "test"
# → state/EMERGENCY_STOP_PRODUCTION ができている

# 2. 状況確認
python3 scripts/emergency_stop.py --status
# → exit 0 + 中身表示

# 3. ガードが効くか（production / dry-run）
python3 scripts/publish.py --slot morning --profile production --date 2026-05-08 --dry-run
# → ⛔ Discord 通知 + sys.exit(0)、Threads API は呼ばれない

# 4. ガードが test を通すか
python3 scripts/publish.py --slot morning --profile test --date 2026-05-08 --dry-run
# → 通常の dry-run 出力（guard を素通り）

# 5. 解除
python3 scripts/emergency_stop.py --off
# → state/EMERGENCY_STOP_PRODUCTION が消えている

# 6. 解除後の production が通るか
python3 scripts/publish.py --slot morning --profile production --date 2026-05-08 --dry-run
# → 通常の dry-run 出力
```

E2E（Discord 経由）：

```
1. Discord で `/緊急停止 動作確認` → Bot が ON 通知を返す
2. `/緊急停止状況` → Bot が ON + 中身を返す
3. （任意）`launchctl kickstart -k gui/$UID/com.example.threads_auto.publish_morning` で発火
   → publish.py が guard で死に、Discord に「⛔ 緊急停止中のため morning 投稿をスキップしました」が来る
4. `/緊急停止解除` → Bot が OFF 通知を返す
5. `/緊急停止状況` → Bot が OFF を返す
```
