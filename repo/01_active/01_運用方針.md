# 01 運用方針（5案生成→3案選択→朝昼晩投稿 / 厚み版）

作成日：2026-05-02

## 全体フロー

```
[毎朝] WebSearchでネタ収集（業界ニュース、トレンド等／8〜10本）
    ↓
[Claude Code] 5つの投稿案を生成（タイプA〜Eから5案、各案は厚み版で書く）
    ↓
[出力] 01_active/posts/YYYY-MM-DD_5案.md に番号付きで提示
    ↓
[OWNER] 番号で選択（例: 「1,3,5」）
       or 修正指示（例: 「2の冒頭をもっと短く」）
       or 却下（例: 「全部見直して」）
    ↓
[投稿]
    7:00 → 承認済み1本目
    12:00 → 承認済み2本目
    20:00 → 承認済み3本目
```

5案それぞれがタイプ別のスレッド数（A/B/D 型は 1〜2 本、C/E 型は 2〜3 本）を持つのが特徴（v3 方針 / 詳細は下記「スレッド数方針 v3」セクション）。

## 投稿タイミング

| 時刻  | 役割                             |
| ----- | -------------------------------- |
| 7:00  | 1本目（朝の通勤・始業前向け）    |
| 12:00 | 2本目（昼休み向け）              |
| 20:00 | 3本目（仕事終わり・views最強帯） |

時刻はランダム化±15分（自動投稿規制対策）。

## 5案の振り分け原則

- 5タイプ（A〜E）から **必ず3タイプ以上をカバー** する5案を出す
- 同じタイプを2案出してもいい（その日の素材次第）
- ただし全部同じタイプはNG（多様性を担保）
- **タイプ別スレッド数の遵守**：A/B/D 型は 1〜2 本、C/E 型は 2〜3 本。3 本超は禁止

## 厚み確保のチェックリスト（5案生成時に各案で確認）

各案を出す前に以下を確認：

- [ ] 機能・論点・観察を **2〜3点に分解** したか
- [ ] ビフォアフを具体化したか（「便利になる」ではなく「〇〇が△△になる」）
- [ ] OWNER経歴を入れる場合、ネタとの相性は良いか（ツール導入/組織導入系のみ）
- [ ] 「前回」「以前」「過去Top1」が入っていないか
- [ ] 編集者の自己解説（「翻訳しました」「整理しました」）が入っていないか
- [ ] 次投稿への予告は「ふんわり」か（具体数字での予告NG）
- [ ] スレッド本数はタイプの推奨範囲内か

## 選択の柔軟性

OWNER側のアクション：

| アクション | 例                                                        | 動作                                       |
| ---------- | --------------------------------------------------------- | ------------------------------------------ |
| 番号選択   | 「1,3,5」                                                 | その3案を朝/昼/晩順に投稿予約              |
| 順序指定   | 「3を朝、1を昼、5を夜」                                   | 指定通りに投稿予約                         |
| 部分修正   | 「2の冒頭短く」                                           | 該当案を修正版に差し替え                   |
| 全部却下   | 「全部見直して」                                          | 5案を再生成                                |
| スキップ   | 「今日休む」                                              | バックログから3案補充                      |
| 厚み調整   | 「3を 2 本→1 本に減らして」「5 を 1 本→2 本に厚くして」 | スレッド本数の指定変更（タイプ別の上限内） |

## レビュー時刻

固定しない。朝のうちに済ませる前提。スマホで完結する想定（実装後回し）。

## 緊急速報差し込み

午後に大きなニュースが出た場合、その日の20時枠を差し替え可（任意）。

## 休日運用

日曜は省略OK。月曜朝の投稿はバックログから補充。

## バックログ運用

- OWNERが「今日休む」を選んだ日のためのストックを最低5本（タイプ別バランスよく）
- v3 のスレッド数方針に従ってストック（A 型単発は OK だが、密度を満たさない単発はストックしない）
- レビュー忘れ・通信障害時の自動補充にも使う

## 自動投稿規制対策

- 投稿時刻に±15分のランダム化
- 文体パターンを5〜8種回す（タイプAでも毎回同じ書き出しにしない）
- 月1〜2回、OWNERの手書き投稿を必ず混ぜる（休む日に手動投稿）

## 2章 v2セットとの実験位置

- 2章 v2：朝昼ペア+夜本命の連動／毎朝レビュー型
- **01_active：5案→3案選択／厚み版** ← このセット

並走させて、どちらが伸びるかで判断する。

## スレッド数方針 v3（2026-05-06 確定・反映済）

3スレッド前提だと中身が薄くなるケースが多発したため、スレッド数の上限を全体的に下げる方針に変更。**02/03/README への反映完了（2026-05-06）**。

### 確定方針

| タイプ                | 新スレッド数                                    |
| --------------------- | ----------------------------------------------- |
| 🅰️ データドリブン型 | 単発 or 2スレ（URL 有時）                       |
| 🅱️ ストーリー型     | 原則 1〜2スレ                                   |
| 🅲️ 解説型           | **2〜3スレ許容**（E型と同じ扱いに格上げ） |
| 🅳️ 比較・選択型     | 原則 1〜2スレ                                   |
| 🅴️ 警告・注意喚起型 | 2〜3スレ許容                                    |

C型を 2〜3スレ許容にしたのは「翻訳する人」ポジションの最大の武器が「用語→比喩→使い時→落とし穴」の4段階分解にあり、これを1スレに圧縮すると差別化要素が落ちるため。

### URL ルール（明文化）

- **URL を入れる場合は最終スレに「決まり文句+URL」**（必ず2スレ以上になる）
- 1スレ目に URL を入れない原則は維持（Threads インプレッション低下対策）
- A型単発は URL なしの場合のみ成立、URL を貼るなら自動的に2スレに
